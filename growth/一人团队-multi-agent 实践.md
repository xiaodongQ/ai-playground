## 📍 项目缘起

### 为什么做这件事？

在 2026-03-08 之前，我一直是一个人单打独斗：
- 写技术博客 → 自己查资料、自己写、自己润色
- 学习新技术 → 自己找教程、自己实践、自己总结
- 理财规划 → 自己研究市场、自己分析、自己决策

**痛点**：
1. 时间碎片化，难以深度专注
2. 角色切换成本高（刚查完资料就要切换成写作模式）
3. 缺少"第二意见"，容易陷入思维盲区

**灵感来源**：
- OpenClaw 的多 Agent 架构设计
- "一人公司"概念（一个人活成一支队伍）
- AI Agent 协作的成熟度已经可以支持实际工作流

**核心想法**：
> 如果我有一个虚拟团队，每个成员都有自己的专业领域，我只需要负责决策和评审，那效率会提升多少？

## 🎯 系统设计目标

### 核心原则

| 原则 | 说明 |
|------|------|
| **专业分工** | 每个 Agent 有明确的职责边界和人设 |
| **统一入口** | 日常对话通过「小黑 - 管家」统一路由 |
| **记忆共享** | 全局记忆由小黑维护，子 Agent 按需读取 |
| **流程透明** | 协作过程可追溯、可复盘 |
| **人类决策** | 关键决策（如博客发布）必须经过用户评审 |

## 🛠️ 实施过程

### 第一阶段：架构设计（2026-03-08）

**关键决策**：

1. **架构选择**：采用独立 workspace 方案
   - 每个 Agent 有独立的配置文件和记忆
   - 小黑 - 管家维护全局记忆（MEMORY.md）
   - 避免 Agent 间直接通信，统一由小黑转发

2. **路由模式**：混合模式
   - 关键词匹配优先（快速、可预测）
   - LLM 语义判断辅助（处理模糊场景）
   - 群聊支持@指定和引用检测

3. **记忆机制**：部分共享
   - 小黑存储全局记忆（用户偏好、项目背景）
   - 子 Agent 有自己的专业记忆
   - 通过文件共享实现跨 Agent 上下文

### 第三阶段：协作流程设计（2026-03-08）

**典型场景**：写一篇技术博客

```
用户请求 → 小黑 - 管家分析意图
              ↓
    ┌─────────┴─────────┐
    ↓                   ↓
Edu·伴学堂          Wri·执笔人
（研究资料）          （撰写文章）
    ↓                   ↓
    └─────────┬─────────┘
              ↓
        小黑 - 管家汇总
              ↓
          用户评审 ←─── 【关键控制点】
              ↓
        小黑 - 管家提交 Git
```

**关键设计**：
- 博客发布前必须经过用户评审（P0 规则）
- 小黑 - 管家负责提交评审，用户确认后才 Git 提交
- 工作流记录到 `memory/workflow-{timestamp}.md` 便于复盘

## 💡 核心洞察

### 1. 人设不是装饰，是行为指南

最初我以为人设只是"让对话更有趣"，但实际配置过程中发现：

**人设决定了**：
- Agent 会如何理解问题（技术匠看到问题想代码，伴学堂想教程）
- Agent 会优先调用什么工具（执笔人用搜索找素材，技术匠用 Claude Code 跑测试）
- Agent 的回答风格（财多多的谨慎 vs 技术匠的严谨）

**教训**：人设描述要具体，包含"你会做什么"和"你不会做什么"。

### 2. 路由规则需要"模糊容忍度"

实践发现的问题：

| 场景 | 预期路由 | 实际问题 | 解决方案 |
|------|---------|---------|---------|
| "帮我学 Python" | Edu | 是要教程还是要代码？ | 路由后由 Agent 反问澄清 |
| "写个爬虫" | Dev | 是要代码还是要学习爬虫原理？ | 关键词权重调整 |
| "分析一下这个基金" | Fin | 是要投资建议还是要技术分析？ | 语义相似度阈值 0.6，低于则提示确认 |

**经验**：
- 关键词匹配要设置优先级
- 模糊场景宁可提示确认，不要猜错
- 允许用户显式指定（@Agent 或直接私聊）

### 3. 记忆共享是协作的关键

**问题场景**：
- 用户跟 Edu 学了 React 基础
- 第二天找 Dev 实现 React 项目
- Dev 不知道用户已经学过基础，从最基础讲起

**解决方案**：
- 小黑维护全局 MEMORY.md，记录学习进度、项目背景
- 子 Agent 启动时读取相关记忆
- 工作流文件记录协作历史

**待优化**：
- 目前需要手动读取记忆，未来可以自动注入相关上下文
- 记忆更新时机需要更智能（不是每次对话都写）

### 4. 人类评审不可省略

**博客发布流程**是我特意设计的"人类在环"（Human-in-the-loop）：

```
AI 撰写 → 用户评审 → AI 修改 → 用户确认 → Git 提交
```

**为什么重要**：
- AI 可能生成事实错误或过时的内容
- 用户需要对发布内容负责
- 评审过程本身是 AI 学习用户偏好的机会

**实现**：
- Wri·执笔人被禁止直接 Git 提交
- 小黑 - 管家负责创建评审请求
- 用户确认后小黑才执行 Git 操作

## ⚠️ 踩坑记录

### 坑 1：路由规则过于严格

**问题**：最初设计路由规则时，试图用关键词完全匹配，结果：
- "我想了解一下区块链" → 没匹配到"学习"关键词，路由失败
- "帮我看看这个代码" → 没匹配到"代码"关键词（实际是"这个代码"）

**解决**：
- 增加同义词库（学习=了解=研究=入门）
- 使用语义相似度而非纯关键词匹配
- 设置阈值，低于阈值则提示确认

### 坑 2：记忆文件权限问题

**问题**：子 Agent 尝试读取全局 MEMORY.md 时，遇到权限问题。

**解决**：
- 统一使用 `/root/.openclaw/workspace/` 作为全局工作区
- 各 Agent workspace 作为私有工作区
- 小黑负责同步共享记忆

### 坑 3：协作流程不透明

**问题**：用户不知道小黑在背后调用了哪些 Agent，感觉像黑盒。

**解决**：
- 创建 `WORKFLOW-TEMPLATE.md` 模板
- 每次协作生成 `memory/workflow-{timestamp}.md`
- 记录参与 Agent、各步骤输入输出、最终结果
