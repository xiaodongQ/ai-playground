## 一、系统概述

### 1.1 设计理念

一人团队 Multi-Agent 系统是基于 **OpenClaw** 框架构建的多智能体协作系统，旨在通过专业化分工和协作编排，为用户提供全方位的个人助理服务。

**核心设计原则**：
- 🎯 **专业化分工** - 每个 Agent 专注特定领域，提供深度服务
- 🔄 **中心化编排** - 小黑 - 管家作为路由中枢，统一调度
- 📝 **部分共享记忆** - 全局记忆 + 专业记忆，平衡共享与隐私
- 🛡️ **安全可控** - 敏感操作需用户确认，日志可追溯

### 1.2 Agent 列表

| Agent | 表情 | 职责 | 触发关键词 |
|-------|------|------|-----------|
| **小黑 - 管家** | 🖤 | 意图识别、路由编排、全局记忆管理 | 通用对话、闲聊、简单查询 |
| **Dev·技术匠** | 🔧 | 编程、架构、代码审查、调试 | 代码、Bug、架构、API、数据库 |
| **Edu·伴学堂** | 📚 | 学习辅导、概念讲解、资源推荐 | 学习、教程、概念、入门、原理 |
| **Wri·执笔人** | ✍️ | 写作、润色、博客、文案 | 写博客、文章、润色、写作 |
| **Fin·财多多** | 💰 | 理财建议、投资分析、预算规划 | 理财、投资、股票、基金、预算 |

### 1.3 技术栈

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **基础框架** | OpenClaw | 多 Agent 编排框架 |
| **通信通道** | 飞书 (Feishu) | IM 机器人集成 |
| **LLM 模型** | bailian/qwen3.5-plus | 默认模型 |
| **代码执行** | Claude Code | Dev Agent 专用 |
| **搜索工具** | Tavily → Exa → Multi-Search | 优先级降级策略 |
| **记忆存储** | Markdown 文件系统 | 简单可维护 |

### 2.2 通信模式

```
用户 → 小黑 → 子 Agent → 小黑 → 用户
          ↑              │
          └──── 记忆 ←───┘
```

**关键特性**：
- Agent 间不直接通信，统一由小黑转发
- 所有交互经过小黑，便于日志和记忆统一
- 每个 Agent 专注自己的领域，无需感知其他 Agent

## 四、路由机制

### 4.1 路由模式：混合模式

采用**关键词优先，LLM 辅助判断**的混合策略。

### 4.2 路由决策树

```
用户输入
   │
   ├─ 显式指定 Agent？（如 @Dev·技术匠） → 直接转交
   │
   ├─ 关键词匹配 ≥2 个？ → 转交对应 Agent
   │
   ├─ 关键词冲突或模糊？ → LLM 意图识别
   │
   ├─ 通用知识/闲聊/简单查询？ → 小黑直接处理
   │
   └─ 混合需求？ → 小黑启动协作流程（串行/并行调度）
```

### 4.3 关键词匹配规则

#### Dev·技术匠 🔧
**强关键词**：代码、编程、Bug、调试、重构、架构、算法、函数、接口、API、数据库、SQL、Git、测试、部署、CI/CD、Docker、前端、后端、DevOps

#### Edu·伴学堂 📚
**强关键词**：学习、教程、入门、概念、原理、是什么、为什么、怎么学、学习路线、学习计划、推荐书籍、课程、资料、理解、解释、说明、讲解

#### Wri·执笔人 ✍️
**强关键词**：写博客、文章、写作、润色、文案、发布、大纲、标题、翻译、改写、优化、编辑、校对、文笔、风格、读者、公众号、知乎、技术文档

#### Fin·财多多 💰
**强关键词**：理财、投资、股票、基金、比特币、加密货币、预算、省钱、财务、收益、风险、定投、资产配置、portfolio、行情、估值、市盈率、收益率

### 4.4 LLM 意图识别

当关键词匹配失败或命中多个类别时，调用 LLM 进行意图分类。

**分类 Prompt 模板**：
```
请分析用户意图，将其分类到以下类别之一：

类别：
- GENERAL: 通用对话、闲聊、简单查询
- DEV: 编程、代码、技术实现
- EDU: 学习、概念理解、知识调研
- WRI: 写作、润色、内容创作
- FIN: 理财、投资、财务规划

用户输入：{user_input}

只返回类别代码，不要解释。
```

### 4.5 群聊消息归属规则（混合模式）

**优先级**：
```
1. 有 @ → 回复指定的人
   ↓
2. 有引用 → 回复引用的消息
   ↓
3. 无@无引用 → 智能判断话题连续性
   ↓
4. 不确定 → 提示用户确认
```

**语义相似度阈值**：
| 相似度 | 处理方式 |
|--------|---------|
| ≥ 0.8 | 高置信度，直接回复 |
| 0.6-0.8 | 中等置信度，直接回复 |
| < 0.6 | 低置信度，提示确认 |

## 五、协作工作流

### 5.1 协作模式

#### 串行协作
```
用户 → 小黑 → Agent1 → 小黑 → Agent2 → 小黑 → Agent3 → 小黑 → 用户
```

**场景**：技术博客写作
1. Edu 生成概念笔记
2. Dev 基于笔记写代码示例
3. Wri 整合成文

#### 并行协作
```
用户 → 小黑 → ┬ → Agent1 → ┐
              ├ → Agent2 → ├ → 小黑 → 用户
              └ → Agent3 → ┘
```

**场景**：多维度分析（同时调研技术、市场、财务信息）

### 5.2 工作流记录

**记录时机**：
- 单 Agent 任务：不记录
- 多 Agent 协作（≥2 个专业 Agent）：必须记录

**记录位置**：`memory/workflow-{YYYY-MM-DD-HHMMSS}.md`

**模板文件**：`WORKFLOW-TEMPLATE.md`

### 5.3 事后总结展示（方案 B）

多 Agent 协作完成后，向用户展示精简摘要：

```
🖤 小黑：任务完成！

【协作过程】
📚 Edu·伴学堂 → 研究量化交易基础（2.5 分钟）
🔧 Dev·技术匠 → 编写示例代码（3 分钟）
✍️ Wri·执笔人 → 整合撰写文章（2 分钟）

【总耗时】7.5 分钟
【产出】博客文章 + 示例代码

【最终结果】
[博客链接]
```

### 5.4 典型协作场景

#### 场景 1：技术博客写作
```
用户："我想写一篇关于 Python 量化交易的博客"
小黑 → Edu·伴学堂：生成量化交易核心概念笔记
小黑 → Dev·技术匠：基于笔记编写示例代码
小黑 → Wri·执笔人：整合所有材料撰写博客
小黑 → 用户：返回最终博客 + 协作过程摘要
```

#### 场景 2：学习 + 实践
```
用户："我想学习 React，并做个待办事项应用"
小黑 → Edu·伴学堂：输出 React 学习路线图
用户学习后 → 小黑 → Dev·技术匠：帮助搭建项目框架
过程中：用户可随时与 Dev/Edu 讨论细节
小黑：保持上下文，跟踪学习进度
```

## 六、记忆系统

### 6.1 模式：部分共享（小黑 - 管家为中心）

小黑 - 管家存储全局记忆，子 Agent 按需读取。

### 6.2 记忆分类

#### 全局记忆（小黑 - 管家存储）
**位置**: `/root/.openclaw/workspace/MEMORY.md`

**内容**：
- 用户基本信息（姓名、称呼、偏好）
- 项目背景与上下文
- 长期目标和计划
- 重要决策记录
- 跨 Agent 协作历史

**访问权限**：所有 Agent 可读，仅小黑可写

#### 专业记忆（各 Agent 独立存储）
**位置**: `/root/.openclaw/workspace-{agent}/memory/`

**内容**：
- 专业领域对话历史
- 领域特定偏好（如代码风格、写作风格）
- 专业项目进度

**访问权限**：仅对应 Agent 可读写

### 6.3 记忆共享流程

#### 小黑 → 子 Agent（传递上下文）
```
用户请求 → 小黑分析 → 需要路由 → 
小黑提取关键上下文 → 传递给目标 Agent → 
目标 Agent 处理 → 返回结果
```

#### 子 Agent → 小黑（更新全局记忆）
```
子 Agent 完成任务 → 发现重要信息 → 
通知小黑更新 MEMORY.md → 小黑确认并写入
```

### 6.4 隐私保护

| 信息类型 | 存储位置 | 访问权限 |
|---------|---------|---------|
| 财务数据 | Fin 独立记忆 | 仅 Fin + 小黑 |
| 代码/项目 | Dev 独立记忆 | Dev + 小黑 |
| 学习笔记 | Edu 独立记忆 | Edu + 小黑 |
| 文章内容 | Wri 独立记忆 | Wri + 小黑 |
| 用户基本信息 | MEMORY.md | 全部 Agent |

### 7.2 规则详情

#### RULE-001: 博客发布评审流程（P0 强制）
```
所有博客提交前必须经过用户评审。

流程：
1. Wri·执笔人 撰写/修改博客
2. 小黑 - 管家 提交评审（展示博客链接 + 预览）
3. 用户确认或提出修改意见
4. 评审通过后，小黑 - 管家 执行 Git 提交

禁止行为：
- ❌ Wri·执笔人 直接执行 git push
- ❌ 小黑 - 管家 未经确认提交
- ❌ 跳过评审流程
```

#### RULE-002: 群聊消息归属混合模式（P0 强制）
```
群聊中消息响应的归属规则（优先级从高到低）：

1. 有 @ → 回复指定的人
   ↓
2. 有引用 → 回复引用的消息
   ↓
3. 无@无引用 → 智能判断话题连续性
   ↓
4. 不确定 → 提示用户确认
```

#### RULE-003: 多 Agent 协作工作流记录（P1 重要）
```
多 Agent 协作任务必须记录完整工作流。

记录时机：
- 单 Agent 任务：不记录
- 多 Agent 协作（≥2 个专业 Agent）：必须记录

记录位置：memory/workflow-{YYYY-MM-DD-HHMMSS}.md
```

#### RULE-004: 搜索工具优先级策略（P1 重要）
```
搜索工具使用优先级：

第一选择：Tavily（AI 优化，简洁结果）
    ↓ API 额度用尽/不可用
第二选择：Exa（代码/技术搜索强项）
    ↓ 需要中文内容/不可用
第三选择：Multi-Search（百度/谷歌等 17 个引擎）
```

#### RULE-005: 挑战式交互原则（P2 建议）
```
在分析和评价类对话中，采用挑战式交互：

1. 多挑战用户 - 不盲目认同，提出不同视角
2. 指出不足 - 直接点出逻辑漏洞、假设问题、盲点
3. 提升方向 - 给出具体可执行的改进建议
4. 第一性原理反问 - 回归本质，问"为什么一定要这样"
```

### 7.3 违规处理

| 优先级 | 说明 | 违规处理 |
|--------|------|---------|
| P0 | 强制规则，必须遵守 | 立即停止，向用户报告 |
| P1 | 重要规则，尽量遵守 | 记录日志，事后报告 |
| P2 | 建议规则，参考执行 | 无需报告 |

## 附录 B：故障排查

### B.1 Agent 无响应

**可能原因**：
- 用户/群聊不在允许列表中
- Gateway 未运行
- 飞书 App Secret 过期

**排查步骤**：
```bash
# 检查 Gateway 状态
openclaw gateway status

# 检查允许列表
cat /root/.openclaw/credentials/feishu-*-allowFrom.json

# 查看日志
tail -f /root/.openclaw/logs/*.log
```

### B.2 路由错误

**可能原因**：
- 关键词匹配不准确
- LLM 意图识别偏差

**解决方案**：
- 调整 `ROUTING.md` 中的关键词列表
- 优化 LLM 分类 Prompt

### B.3 记忆丢失

**可能原因**：
- 文件未保存
- 工作区切换错误

**解决方案**：
- 确保写入 `MEMORY.md` 而非临时文件
- 子 Agent 通过小黑更新全局记忆
