# 01-项目调研 - SeaweedFS 核心概念和特性

> 🌊 深入理解 SeaweedFS 的设计哲学和核心架构

## 1.1 项目定位

### 1.1.1 什么是 SeaweedFS？

**SeaweedFS** 是一个简单且高度可扩展的分布式文件系统，专为以下两个核心目标设计：

1. **存储数十亿文件** - 支持海量小文件高效存储
2. **快速服务文件** - O(1) 磁盘寻址，单次读取

SeaweedFS 始于实现 [Facebook Haystack 设计论文](http://www.usenix.org/event/osdi10/tech/full_papers/Beaver.pdf)，并融合了以下系统的思想：
- [Facebook f4: Warm BLOB Storage System](https://www.usenix.org/system/files/conference/osdi14/osdi14-paper-muralidhar.pdf) - 纠删码
- [Facebook Tectonic Filesystem](https://www.usenix.org/system/files/fast21-pan.pdf) - 分布式元数据
- [Google Colossus File System](https://cloud.google.com/blog/products/storage-data-transfer/a-peek-behind-colossus-googles-file-system) - 云原生存储

### 1.1.2 核心设计理念

```
┌─────────────────────────────────────────────────────────────┐
│                    设计哲学                                  │
├─────────────────────────────────────────────────────────────┤
│  • 简单至上 - 架构清晰，易于理解和运维                       │
│  • 性能优先 - O(1) 磁盘读取，40 字节元数据开销                │
│  • 水平扩展 - 添加服务器即可扩容，无需数据重平衡             │
│  • 云原生 - 本地热数据 + 云端温数据混合存储                  │
│  • 多协议 - S3、WebDAV、Hadoop、FUSE、Iceberg 统一支持      │
└─────────────────────────────────────────────────────────────┘
```

### 1.1.3 与传统的区别

| 传统分布式存储 | SeaweedFS |
|--------------|-----------|
| 中央元数据服务器管理所有文件 | 中央 Master 仅管理 Volume，Volume Server 管理文件元数据 |
| 元数据服务器成为瓶颈 | 元数据分散，并发压力分散 |
| 小文件存储效率低 | 专为小文件优化，40 字节/文件元数据 |
| 扩容需要数据重平衡 | 添加服务器即可，无需重平衡 |

## 1.2 核心架构组件

### 1.2.1 三层架构

```
┌────────────────────────────────────────────────────────────────┐
│                       接入层 (Gateway)                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐       │

│  │ Gateway  │  │ Gateway  │  │   SDK    │  │  Mount   │       │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘       │
└────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────────┐
│                       元数据层 (Filer)                          │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Filer (Stateless) - 支持多种后端存储                     │  │
│  │  MySQL, Postgres, Redis, Cassandra, HBase, MongoDB...    │  │
│  └──────────────────────────────────────────────────────────┘  │

                              │

┌────────────────────────────────────────────────────────────────┐
│                       存储层 (Storage)                          │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐               │
│  │  Master  │────▶│  Volume  │◀────│  Volume  │               │
│  │ Server   │     │ Server 1 │     │ Server 2 │               │
│  │  :9333   │     │  :9340   │     │  :9341   │               │
│  └──────────┘     └──────────┘     └──────────┘               │
│       │                │                  │                    │
│       │                ▼                  ▼                    │
│       │          ┌──────────┐      ┌──────────┐               │
│       │          │ Volume   │      │ Volume   │               │
│       │          │   .dat   │      │   .dat   │               │
│       │          │   .idx   │      │   .idx   │               │
│       │          └──────────┘      └──────────┘               │
└────────────────────────────────────────────────────────────────┘

### 1.2.2 Master Server

**职责**：
- 管理所有 Volume Server 和 Volume
- 分配 Volume ID 给写入请求
- 维护 Volume 到 Volume Server 的映射关系
- 处理 Volume 复制和故障恢复

**特性**：
- 支持 Raft 协议实现高可用
- 自动故障转移，无单点故障
- 内存存储元数据，启动时从磁盘恢复

**端口**：
- HTTP: 9333（默认）
- gRPC: 9333 + 10000 = 19333（默认）

### 1.2.3 Volume Server

**职责**：
- 存储实际的文件数据（Blobs）
- 管理 Volume 内的文件元数据
- 处理文件读写请求
- 支持复制和纠删码

**核心设计**：
- 每个 Volume 是一个大文件（.dat + .idx）
- .dat 文件存储实际数据
- .idx 文件存储偏移量索引
- O(1) 磁盘读取：通过 Volume ID + File ID 直接定位

**特性**：
- 支持多磁盘、多目录
- 自动压缩和整理碎片
- 支持 TTL 自动过期
- 支持图片缩放

**端口**：
- HTTP: 9340（默认）
- gRPC: 9340 + 10000 = 19340（默认）

### 1.2.4 Filer

**职责**：
- 提供 POSIX 风格的目录和文件抽象
- 管理文件路径到 File ID 的映射
- 存储文件元数据（权限、时间戳等）

**特性**：
- 无状态设计，可水平扩展
- 支持多种元数据存储后端：
  - 关系型：MySQL、Postgres、TiDB、CockroachDB
  - NoSQL：Redis、Cassandra、HBase、MongoDB
  - 嵌入式：Sqlite、LevelDB、RocksDB
  - 搜索：ElasticSearch

**端口**：
- HTTP: 8888（默认）
- gRPC: 8888 + 10000 = 18888（默认）

### 1.2.5 S3 Gateway

**职责**：
- 实现 Amazon S3 API 兼容层
- 将 S3 请求转换为 SeaweedFS 内部操作
- 管理 IAM 用户和策略

**支持的 S3 API**：
- Bucket 操作：ListBuckets, CreateBucket, DeleteBucket, etc.
- Object 操作：PutObject, GetObject, DeleteObject, ListObjects, etc.
- Multipart Upload: CreateMultipartUpload, UploadPart, CompleteMultipartUpload
- 预签名 URL：支持临时访问授权

**端口**：
- HTTP: 8333（默认）

## 1.3 数据存储原理

### 1.3.1 Volume 结构

```
Volume 文件结构:
┌─────────────────────────────────────────┐
│            .dat 文件（数据）             │
├─────────────────────────────────────────┤
│  ┌───────────┐  ┌───────────┐          │
│  │   File 1  │  │   File 2  │  ...     │
│  │  Header   │  │  Header   │          │
│  │  + Data   │  │  + Data   │          │
│  └───────────┘  └───────────┘          │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│            .idx 文件（索引）             │
├─────────────────────────────────────────┤
│  FileID → Offset + Size + Cookie        │
│  FileID → Offset + Size + Cookie        │
│  ...                                    │
└─────────────────────────────────────────┘
```

### 1.3.2 文件定位流程

```
Client 请求：VolumeID + FileID
                    │
                    ▼
         ┌──────────────────┐
         │   Master Server  │
         │  查询 Volume 位置  │
         └────────┬─────────┘
                  │
                  ▼
    Volume Server: http://vs-host:9340/123/456
                  │
                  ▼
         ┌──────────────────┐
         │   .idx 文件查找   │
         │  FileID → Offset │
         └────────┬─────────┘
                  │
                  ▼
         ┌──────────────────┐
         │   .dat 文件读取   │
         │  seek(offset)    │
         │  read(size)      │
         └────────┬─────────┘
                  │
                  ▼
              返回数据
```

### 1.3.3 元数据开销

| 组件 | 开销 | 说明 |
|------|------|------|
| Master Server | ~100 字节/Volume | 仅管理 Volume 级别元数据 |
| Volume Server | 40 字节/文件 | 存储在 .idx 文件中 |
| Filer | ~200 字节/文件 | 路径、权限、时间戳等 |

**对比**：传统文件系统每文件元数据开销通常为 KB 级别

## 1.4 数据分布和复制

### 1.4.1 复制策略

SeaweedFS 支持多种复制级别：

```
复制类型：
├── 000 - 无复制
├── 001 - 1 副本（默认）
├── 010 - 2 副本
├── 100 - 3 副本
├── 110 - 2 副本 + 跨机架
├── 200 - 3 副本 + 跨数据中心
└── ... 更多组合
```

### 1.4.2 机架和数据中心感知

```
数据中心 A
├── 机架 1
│   ├── Volume Server 1
│   └── Volume Server 2
└── 机架 2
    ├── Volume Server 3
    └── Volume Server 4

数据中心 B
├── 机架 3
│   ├── Volume Server 5
│   └── Volume Server 6
└── 机架 4
    ├── Volume Server 7
    └── Volume Server 8

复制策略：200（3 副本跨数据中心）
- 副本 1: 数据中心 A, 机架 1
- 副本 2: 数据中心 A, 机架 2
- 副本 3: 数据中心 B, 机架 3
```

### 1.4.3 纠删码 (Erasure Coding)

SeaweedFS 实现了类似 Facebook f4 的纠删码：

```
原始数据：20 个数据块
校验数据：4 个校验块
存储开销：24/20 = 1.2x
容忍故障：任意 4 个块丢失

优势：
- 相比 3 副本（3x 开销），节省 60% 存储
- 适合温数据和冷数据
- 可配置 EC 比例（如 10+3, 14+5, 20+4）
```

## 1.6 性能特性

### 1.6.1 基准测试数据

基于官方和社区测试数据：

| 场景 | 配置 | 性能 |
|------|------|------|
| **小文件写入** | 1KB 文件，3 副本 | ~50K ops/s 单节点 |
| **小文件读取** | 1KB 文件，内存缓存 | ~100K ops/s 单节点 |
| **大文件吞吐** | 100MB 文件 | ~500 MB/s 单节点 |
| **并发连接** | 标准配置 | ~10K 并发连接 |

### 1.6.2 性能优化点

1. **O(1) 磁盘读取** - 直接定位，无需遍历目录
2. **内存索引** - .idx 文件加载到内存
3. **连接复用** - HTTP Keep-Alive
4. **批量操作** - 支持批量读写
5. **自动压缩** - 根据 MIME 类型自动压缩

### 1.7.2 自修复存储

```
正常流程：

- 防止静默数据损坏
- 应对意外宕机

## 1.8 部署模式

### 1.8.1 开发模式（weed mini）

```bash
./weed mini -dir=/data
```

**特点**：
- 单进程启动所有组件
- 自动配置优化
- 适合开发、测试、学习

### 1.8.2 生产模式（weed server）

```bash
# Master
weed master -mdir=/data/master -ip=master-host -port=9333

# Volume Server（每台存储节点）
weed volume -dir=/data/volume1 -master=master-host:9333 -port=9340

# Filer
weed filer -master=master-host:9333 -ip=filer-host -port=8888

# S3 Gateway
weed s3 -filer=filer-host:8888 -ip=s3-host -port=8333
```

**特点**：
- 组件独立部署
- 手动配置优化
- 适合生产环境

### 1.8.3 Kubernetes 部署

```yaml
# 使用 Helm Chart 或 Operator
helm repo add seaweedfs https://seaweedfs.github.io/seaweedfs/helm
helm install seaweedfs seaweedfs/seaweedfs
```

## 1.9 小结

### 核心要点

1. **架构简洁** - Master + Volume + Filer 三层，职责清晰
2. **性能优异** - O(1) 磁盘读取，40 字节/文件元数据
3. **高度扩展** - 支持数千节点，PB 级容量
4. **S3 兼容** - 无缝对接现有 S3 生态
5. **云原生** - 混合云存储，成本优化

### 适用场景

✅ **适合**：
- 海量小文件存储（图片、视频缩略图、日志）
- 需要 S3 兼容 API 的场景
- 混合云存储需求
- 需要水平扩展的场景

❌ **不适合**：
- 需要强一致性事务的场景
- 频繁随机写入大文件的场景
- 需要 POSIX 完全兼容的场景（虽然有 FUSE，但非 100% 兼容）
